Problem

Utility document intelligence requires extracting 12+ entity types (meter IDs, service addresses, tariff codes, billing periods, consumption figures) from structurally inconsistent PDFs at scale. Routing every document to a frontier model (GPT-4-class) achieves high accuracy but at prohibitive cost. Routing to cheaper models reduces cost but introduces silent extraction failures — errors that pass without surfacing. The challenge: build a routing system that learns cost-vs-accuracy trade-offs dynamically and fails loudly rather than silently.

Architecture

POPPER is a three-layer system. The extraction layer runs a model pool — lightweight models (low cost, lower accuracy) and frontier models (high cost, high accuracy) — against each document in parallel up to a configurable depth. The assertion layer applies deterministic post-extraction checks (type validation, cross-field consistency, format constraints) to every extraction result; failure is explicit and structured. The routing layer is a multi-armed bandit RL policy that observes assertion outcomes and updates model-selection weights per entity type, converging on the cheapest model that clears assertions reliably.

Frontier escalation is triggered only on deterministic assertion failure — not on confidence thresholds, which are soft and gameable. This is the core design decision that makes the cost reduction defensible.

Implementation — Key Components

Multi-Armed Bandit Router

UCB1-variant bandit with per-entity-type arm weights. Reward signal is binary: assertion pass = positive reward, assertion fail = zero reward with frontier escalation triggered. Policy converged within ~200 episodes across the production document corpus.

Deterministic Assertion Engine

Rule-based post-extraction validator covering: type constraints (numeric ranges, date formats, regex patterns), cross-field consistency (billing period ↔ consumption figure coherence), and completeness checks (required entity presence per document class). Assertion failure is structured — fails with entity ID and constraint violated, not a generic error.

Model Pool Abstraction

Unified extraction interface over heterogeneous model backends — local quantised models, mid-tier API models, and frontier API models — with consistent output schema enforced at the interface boundary. Adding a new model to the pool requires implementing one method; no changes to the bandit or assertion layers.

IIT Delhi Research Collaboration

Invited to the IIT Delhi Winter Research Residency (Dec 2025) by Prof. Abhilash Jindal's CSE group — the academic authors of the POPPER dataflow system. Studied the internal bandit-based query optimizer architecture; participated in research discussions on extending POPPER to handle data drift and LLM-based pipeline scenarios.
