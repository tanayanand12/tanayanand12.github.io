Contribution Claim

Existing biomedical literature tools (PubMed search, citation managers) are single-step: query → result list. They do not decompose queries, rerank by semantic relevance, construct knowledge graphs across controlled vocabularies, or synthesise citation-backed answers. MRA is a full-pipeline agentic system that does all of these in sequence, reducing literature synthesis from hours to under 5 minutes per query while maintaining full citation traceability.

Architecture — Six-Agent Pipeline

Agent 1

Query Decomposition Agent

Breaks complex biomedical queries into sub-questions across disease mechanisms, treatment modalities, and clinical outcomes.

Agent 2

Parallelised Retrieval Agent

Concurrent PubMed and ClinicalTrials.gov retrieval via ThreadPooling; handles pagination and rate limits transparently.

Agent 3

FAISS Chunk Reranking Agent

Semantic reranking of retrieved chunks against the original query; filters low-relevance retrievals before downstream processing.

Agent 4

Knowledge Graph Construction Agent

Builds entity-relationship graphs using UMLS, SNOMED-CT, Gene Ontology, and KEGG; enables multi-hop reasoning across biomedical ontologies.

Agent 5

AMA Citation Synthesis Agent

Assembles ranked, deduplicated evidence into AMA-formatted citation-backed responses.

Agent 6

WebSocket Query Cancellation Agent

Real-time query cancellation via WebSocket; prevents wasted compute on abandoned queries.

Dataset / Corpus

PubMed (100K+ papers indexed), ClinicalTrials.gov (live retrieval). Controlled vocabularies: UMLS, SNOMED-CT, Gene Ontology, KEGG. No proprietary dataset — all sources are publicly accessible APIs.

Limitations

  • No formal evaluation against a held-out benchmark (BioASQ, MedQA) yet — planned for the paper submission.
  • Knowledge graph construction quality depends on UMLS/SNOMED-CT coverage; rare diseases and novel entities may have sparse ontology links.
  • LLM synthesis layer is not domain-fine-tuned — uses general-purpose GPT; BioMistral fine-tune on COMAP dataset is planned.
  • Real-time performance depends on PubMed API availability; no offline fallback currently implemented.
